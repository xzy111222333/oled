# iOS 全局护眼工具（OLED Saver）产品需求文档 (PRD) v1.0

## 1. 项目背景与立项初衷

### 1.1 痛点分析（为什么要做？）
*   **硬件缺陷（PWM 调光）：** 目前主流 iPhone（如 iPhone X/11/12/13/14/15 系列）均采用 OLED 屏幕。OLED 屏幕在低亮度下普遍采用 **低频 PWM 调光**（频闪）。这就好比一个每秒闪烁几百次的闪光灯对着眼睛照。
*   **生理反应：** 这种频闪会导致用户在夜间或暗光环境下使用手机时，出现严重的眼睛干涩、流泪、视力模糊、偏头痛甚至恶心（俗称“瞎眼”）。
*   **系统局限：** 苹果 iOS 系统自带的“夜览（Night Shift）”仅调节色温（变黄），无法解决频闪问题；“降低白点值”藏在深层设置里，操作繁琐且无法从根本上模拟“类 DC 调光”的效果。
*   **竞品缺失：** 安卓端有成熟的“护眼宝”、“闪烁保护”等应用，通过软件遮罩降低亮度，从而让屏幕硬件维持在高亮度（高频 PWM 或 DC 模式），但 iOS 端缺乏同类简单直接、且符合用户直觉的产品。

### 1.2 产品目的
开发一款 iOS 端的全局护眼应用，核心目的是**“防蓝光 + 防频闪”**。
通过在屏幕最上层覆盖一层半透明的黑色/黄色滤镜，模拟安卓“护眼宝”的效果，强制让用户将系统物理亮度拉高（减少频闪），而通过软件滤镜将视觉亮度压低，从而达到护眼目的。

---

## 2. 竞品参考：安卓“护眼宝”核心功能全集

开发人员需参考以下安卓端成熟功能，但需根据 iOS 限制进行适配：

1.  **全局滤镜覆盖：** 无论在桌面、看视频、玩游戏，滤镜始终覆盖在最上层。
2.  **色温调节（防蓝光）：** 提供暖色（黄/红）滤镜，过滤蓝光。
3.  **亮度调节（防频闪）：** 提供黑色遮罩滤镜，允许屏幕亮度低于系统最低值。
4.  **智能护眼：** 根据时间自动开启/关闭（日落开启，日出关闭）。
5.  **极简操作：** 一个开关，两个滑块（颜色、亮度）。
6.  **通知栏快捷操作：** 在通知中心常驻，方便随时调节（iOS 需考虑替代方案，如控制中心或灵动岛）。

## 3. 核心功能需求 (Must-Have)

**设计原则：** 简单、粗暴、有效。拒绝任何基于 APP 场景的自动化判断。

### 3.1 全局遮罩滤镜 (Core Feature)
*   **功能描述：** APP 启动后，在屏幕最顶层生成一个全屏的 Overlay（遮罩层）。
*   **技术要求：** 遮罩层必须能够穿透点击（Pass-through），即用户开启护眼模式后，不影响对手机其他 APP 的正常触摸操作。

### 3.2 颜色与亮度调节 (UI Controls)
*   **色温滑块：** 调节遮罩层的 RGB 值。从“原色”到“深黄/琥珀色”。
*   **暗度滑块：** 调节遮罩层的 Alpha 值（透明度）。
    *   0%：完全透明（无效果）。
    *   80%：极暗（适合被窝玩手机）。
*   **数值显示：** 滑块旁需显示当前百分比。

### 3.3 “生死绑定”机制 (关键需求 - Kill Switch)
*   **需求描述：** **APP 存活 = 滤镜存在；APP 死亡 = 滤镜消失。**
*   **场景举例：** 用户正在被窝里用护眼模式刷视频，突然想拍一张夜景照片。用户直接上滑杀掉护眼 APP 的后台，屏幕颜色必须**瞬间**恢复到系统默认状态，没有任何延迟，没有任何残留。
*   **技术逻辑：**
    *   滤镜的生命周期必须严格绑定 APP 的进程生命周期。
    *   **禁止**修改系统永久性设置（如不要去改系统的“降低白点值”设置，因为那个杀后台不会恢复）。
    *   必须利用 iOS 的 `applicationWillTerminate` 监听，或者利用 Overlay 本身依附于 APP 进程的特性，确保进程结束时，内存释放，遮罩层自然消失。

### 3.4 极简主界面
*   **大开关按钮：** 屏幕中央一个巨大的“开启/关闭”按钮。
*   **状态反馈：** 开启时按钮发光或变色。

1.  **基于 APP 的场景化自动化：** 不做“打开微信自动变黄”、“打开相册自动关闭”的功能。逻辑复杂且不稳定。
2.  **健康报告/用眼统计：** 不需要统计用户看了多久手机，不需要生成图表。
3.  **眼保健操/训练：** 不需要。
4.  **广告推送：** 界面必须干净，无开屏广告。

## 5. 技术实现与上架策略 (致开发人员)

*注意：iOS 的沙盒机制比安卓严格，实现“全局遮罩”是本项目的最大难点。*

### 5.1 技术路径建议
由于 App Store 严禁常规 APP 在其他 APP 之上绘制 UI（悬浮窗限制），建议尝试以下路径之一，并优先考虑能过审核的方案：

*   **方案 A（画中画/悬浮窗漏洞利用 - 风险高但效果好）：** 利用 iOS 的“画中画” (PiP) 接口或者特殊的 `UIWindow` 层级技巧，生成一个透明的、全屏的、不可交互的视频层或 Window 层覆盖在顶端。
    *   *优点：* 真正模拟安卓体验，杀后台即消失。
    *   *缺点：* 极大概率被 App Store 拒审，需要进行代码混淆或伪装（如伪装成视频播放器）。

*   **方案 B（色彩滤镜辅助功能 - 官方合规路径）：**
    *   APP 不直接绘制遮罩，而是作为一个“快捷开关”。
    *   通过引导用户授权 **Accessibility (辅助功能)** 权限。
    *   APP 内部调节实际上是修改系统的“色彩滤镜”或“缩放滤镜（低光）”参数。
    *   **关于“杀后台恢复”的特殊处理：** 如果采用此方案，必须在 `applicationWillTerminate` (APP 即将被杀) 的生命周期方法中，写入代码**强制关闭**系统的辅助功能开关。确保用户划掉 APP 时，系统设置被还原。

### 5.2 性能要求
*   **CPU 占用：** 滤镜开启时，CPU 占用率需低于 1%。
*   **耗电量：** 必须极低，不能因为开个遮罩导致手机发烫。
*   **触摸延迟：** 开启滤镜后，屏幕触摸延迟增加不能超过 5ms。

## 6. 验收标准 (UAT)

1.  **安装运行：** 在 iPhone (OLED 机型) 上成功运行。
2.  **遮罩测试：** 开启后，打开微信、浏览器、抖音，屏幕明显变暗/变黄，且能正常刷视频、点赞。
3.  **截图测试：** (如果是方案 A) 物理截图时，截图内容应该是变色的（说明是软件层遮罩）；(如果是方案 B) 截图可能是原色。**以肉眼看到的屏幕变暗为准。**
4.  **“杀后台”测试（核心）：**
    *   步骤：开启护眼模式 -> 屏幕变黄 -> 进入多任务界面 -> 上滑关闭本 APP。
    *   结果：手指松开的瞬间，屏幕颜色**立即**恢复正常白亮状态。
5.  **稳定性：** 连续运行 24 小时无闪退。

# iOS OLED 全局护眼软件产品开发说明书 (MVP 版)

## 1. 项目背景与核心目标

### 1.1 项目背景
*   **硬件痛点**：当前 iPhone (iPhone X 及后续机型) 普遍采用 OLED 屏幕。OLED 屏幕在低亮度下（通常低于 25%-30%）采用**低频 PWM 调光**（频闪），导致用户视疲劳、近视加深、散光加重。此外，OLED 自发光特性导致对比度过高、蓝光峰值高，进一步伤害视力。
*   **系统限制 (iOS vs Android)**：
    *   **Android**：允许应用通过 `SYSTEM_ALERT_WINDOW` 权限绘制全局悬浮窗（Overlay），通过覆盖半透明色层实现滤光和变暗。**特点**：应用被杀后台，悬浮窗消失，屏幕瞬间恢复原状。
    *   **iOS**：严格的沙盒机制，**禁止**第三方应用在其他应用（如微信、QQ）之上绘制任何图层。无法实现 Android 式的“悬浮窗滤镜”。
    *   **解决方案**：iOS 端必须通过调用系统底层的**辅助功能 (Accessibility)** 和 **快捷指令 (Shortcuts/App Intents)** 来修改系统全局显示参数（亮度、白点值、色彩滤镜），从而实现全局护眼。
*   **开发环境**：开发者使用 Windows 电脑，采用 Flutter 跨平台框架开发，通过云 Mac 进行 iOS 打包上架。

### 1.2 核心目标
开发一款能上架 Apple App Store 的 iOS 应用，在不越狱、不违反苹果规则的前提下，实现类似安卓“护眼宝”的**全局屏幕调节功能**。核心解决 OLED 频闪、蓝光、高对比度问题，并提供**一键瞬间恢复**屏幕原色的能力。

## 3. iOS 技术边界与合规性声明 (开发者必读)

### 3.1 不可触碰的红线 (会导致拒审或无法实现)
1.  **禁止全局悬浮窗**：不要尝试在 iOS 上绘制全局半透明层，这在非越狱设备上不可能实现。
2.  **禁止私有 API**：不得使用未公开的系统接口控制屏幕驱动。
3.  **禁止医疗宣传**：界面和描述中不得出现“治疗近视”、“降低度数”、“消除频闪”等医疗用语。应使用“缓解视疲劳”、“优化显示效果”、“规避频闪区间”。
4.  **禁止后台常驻修改亮度**：iOS 不允许应用在后台持续修改 `UIScreen.brightness`。亮度锁定需通过用户配置自动化或前台保持实现。

### 3.2 可行的技术路径 (唯一合法全局方案)
1.  **App Intents / 快捷指令 URL Scheme**：用于触发系统设置更改。
2.  **辅助功能 (Accessibility)**：
    *   `Reduce White Point` (降低白点值)：核心用于在不降低硬件亮度的情况下变暗屏幕。
    *   `Color Filters` (色彩滤镜)：核心用于实现红光、黄光、绿光全局过滤。
    *   `Zoom - Low Light` (缩放 - 弱光滤镜)：核心用于实现极暗环境下的额外变暗。
3.  **Widget / 控制中心 (Control Center)**：用于提供“一键恢复”和“模式切换”的入口，解决 iOS 无法杀后台恢复的问题。

## 4. 产品功能详细规格 (MVP 版本)

### 4.1 核心功能一：OLED 防频闪亮度引擎 (全局)
*   **用户界面**：APP 首页提供一个 0-100% 的“视觉亮度滑块”。
*   **底层逻辑 (核心算法)**：
    *   **滑块 0% - 50% 区间**：
        *   系统硬件亮度：从 100% 线性降至 35% (安全阈值)。
        *   降低白点值：保持 0%。
        *   *目的*：在安全范围内尽量使用硬件调光，保证画质。
    *   **滑块 50% - 100% 区间**：
        *   系统硬件亮度：**锁定在 35%** (绝不低于此值，避开低频 PWM 频闪区)。
        *   降低白点值：从 0% 线性拉升至 100%。
        *   *目的*：硬件保持无频闪高亮，通过软件白点值压暗视觉亮度，实现“暗而不闪”。
*   **生效范围**：全局 (微信、QQ、系统界面)。
*   **实现方式**：用户滑动后，点击“应用”，APP 调用快捷指令一次性设置这两个参数。

### 4.2 核心功能二：医疗级全局色调过滤 (全局)
*   **用户界面**：提供 4 个大按钮卡片，对应安卓护眼宝模式。
    1.  **助眠红 (红色模式)**：
        *   *配置*：开启色彩滤镜 (色调选红，强度 100%) + 降低白点值 80%。
        *   *用途*：睡前使用，屏蔽蓝光，促进褪黑素。
    2.  **阅读黄 (小说模式)**：
        *   *配置*：开启色彩滤镜 (色调选黄/暖，强度 60%) + 降低白点值 40%。
        *   *用途*：日常阅读，模拟纸张，防蓝光。
    3.  **散光绿 (绿色模式)**：
        *   *配置*：开启色彩滤镜 (色调选绿，强度 30%) + 关闭深色模式。
        *   *用途*：降低对比度，缓解散光重影。
    4.  **极暗夜 (黑色模式)**：
        *   *配置*：降低白点值 100% + 开启“缩放 - 弱光滤镜”。
        *   *用途*：全黑环境下使用，极致暗光。
*   **生效范围**：全局 (微信、QQ、系统界面)。
*   **实现方式**：点击按钮 -> 调用对应快捷指令 -> 系统参数瞬间变更。

### 4.3 核心功能三：一键瞬间恢复 (全局)
*   **痛点解决**：安卓杀后台即恢复，iOS 设置会持久化。用户拍照或需要真实色彩时需瞬间恢复。
*   **功能定义**：提供一个**“恢复原屏”**的极速入口。
*   **实现方式 (三重重保障)**：
    1.  **APP 内首页大按钮**：显著位置放置“关闭护眼/恢复原色”按钮，点击即重置所有辅助功能。
    2.  **iOS 控制中心小组件 (Control Center Widget)**：引导用户将本软件的“恢复开关”添加到 iOS 控制中心。用户在任何界面（包括相机取景框）下拉控制中心，点一下即可恢复。
    3.  **锁屏小组件 (Lock Screen Widget)**：提供锁屏一键恢复按钮。
*   **逻辑**：点击恢复 -> 调用快捷指令 -> 关闭色彩滤镜、关闭降低白点值、关闭弱光滤镜、恢复自动亮度。

### 4.4 核心功能四：自动化配置 (非强制，仅配置)
*   **说明**：不开发复杂的后台监听逻辑，而是提供“配置包”。
*   **功能**：APP 内提供“导入自动化”按钮。
*   **逻辑**：用户点击后，跳转到 iOS 快捷指令 APP，自动生成一条自动化规则。
    *   *例如*：当时间 23:00 -> 运行“助眠红”指令。
    *   *例如*：当时间 07:00 -> 运行“恢复原屏”指令。
*   **注意**：此功能仅为辅助，不作为核心卖点，核心卖点仍是手动一键切换。

## 6. 技术实现方案 (Flutter + iOS 原生交互)

### 6.1 开发环境
*   **本地**：Windows 10/11 + VS Code + Flutter SDK。
*   **打包**：租赁云 Mac (MacinCloud 等) + Xcode + Apple Developer Account ($99/年)。

#### **一、 项目背景与核心理念**

**1. 项目背景：OLED屏幕的“健康危机”与系统限制**
随着iPhone全系转向OLED屏幕，其技术特性带来了两大核心健康隐患：
*   **低频PWM调光**：在低亮度（通常低于25%-30%）下，OLED屏幕采用低频脉冲宽度调制（PWM）来控制亮度，导致屏幕以约480Hz的频率快速亮灭。虽肉眼难察，但人眼睫状肌会持续受到刺激，引发视疲劳、眼干、酸胀，长期可能加剧近视和散光。
*   **高对比度与蓝光峰值**：OLED的像素自发光特性造就了极致的纯黑与高亮，形成超高对比度，易引发眩光和视疲劳；同时其蓝光光谱峰值较高，可能影响睡眠节律。

**iOS系统的严格限制**：苹果出于安全、隐私和性能考虑，采用沙盒机制，禁止第三方应用在系统底层或其它应用（如微信、QQ）之上绘制任何形式的覆盖层（Overlay）。这直接排除了安卓平台上“护眼宝”类软件通过全局悬浮窗实现滤光、调色的核心技术路径。

**2. 项目目的与理念**
本项目旨在开发一款 **“智能显示参数调节器”** ，而非简单的“滤镜应用”。核心理念是：
*   **不新增，只调度**：不尝试突破iOS限制去绘制滤镜，而是深度挖掘并智能化调度苹果系统**已有**的、全局生效的底层显示参数。
*   **算法驱动体验**：将复杂的OLED护眼医学原理（如PWM安全阈值、蓝光波段）封装为用户无感知的智能算法，提供“一键”或“一滑”的极简体验。
*   **对标“护眼宝”体验，超越其局限**：在合规前提下，提供媲美安卓“护眼宝”的核心调节自由度，并利用iOS的快捷指令与自动化能力，实现更稳定、无感的护眼状态切换。
