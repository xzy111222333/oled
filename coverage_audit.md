# OLEDGuard 需求对照与当前实现状态

## 1. 对照范围

本文件用于把以下三类需求拉到同一张表上：

- `final_plan.md` 中的最终方案
- `聊天记录.md` 中多次重复强调的硬需求
- 安卓护眼宝页面所体现的核心交互结构

目标不是“报喜”，而是明确当前仓库已经实现了什么，还缺什么，哪些属于 iOS 公开能力边界。

---

## 2. 当前已实现

### 2.1 页面与交互

- 已重做为 `单页主控台`
- 首页已包含：
  - 自动亮度开关
  - 亮度滑块
  - 防蓝光过滤度滑块
  - 生物钟开关
  - 自动夜间助眠开关
  - 五种模式切换
  - 一键恢复
  - 护眼评分卡
  - 今日使用时长
  - 疲劳使用时长

- 设置页已包含：
  - 全局桥接状态
  - 系统能力建议动作
  - 建议值查看

### 2.2 核心模型

- 已实现 `过滤度 0-100`
- 已实现 `五模式`
- 已实现 `亮度手动/自动状态`
- 已实现 `PWM 安全阈值算法`
- 已实现 `过滤度 -> 色偏增强 -> 白点压制建议` 的统一模型
- 已实现 `生物钟阶段` 计算
- 已实现 `自动化规则` 数据结构

### 2.3 工程结构

- 已创建原生 iOS 工程描述 `project.yml`
- 已创建主 App、Widget、App Intents、Tests 目标
- 已创建共享状态存储
- 已创建 Widget 快捷入口
- 已创建 Intent 预设动作

---

## 3. 当前未完成

### 3.1 真正的全局系统能力接管

当前仓库还没有真实完成以下全局能力：

- 白点值精确写入
- 色彩滤镜精确写入
- 弱光滤镜/缩放低光
- 深色模式联动
- 控制中心级一键全局切换链路
- 系统自动化触发后的真机执行闭环

原因不是前端没做，而是这些能力需要在真机、系统桥接和审核边界下继续验证，不能在当前 Windows 环境里凭空宣称“已完成”。

### 3.2 上线必需物料

以下仍未完成：

- `.xcodeproj` 生成与 Xcode 编译验证
- 真机运行
- Bundle ID / Team / 签名
- App Group 真值配置
- 图标与商店资源
- 隐私文案
- 审核说明
- TestFlight 测试

---

## 4. 能力分级

### 4.1 当前已原生实现

- 前台亮度控制
- 过滤度算法计算
- 模式切换
- 自动化规则存储
- Intent 与 Widget 的状态联动

### 4.2 需要系统桥接

- 降低白点值
- 色彩滤镜
- 极暗弱光

### 4.3 不应作为当前公开路线核心承诺

- 原彩显示直接控制
- 夜览强度直接控制
- 越权式后台全局覆盖
- 私有 API 控制底层显示驱动

---

## 5. 结论

当前仓库已经从“文档/骨架”推进到了“产品主界面 + 算法层 + 工程目标结构 + 能力桥接模型”。

但它仍然不是最终可上线包体。

如果后续继续推进，正确路径不是再写空页面，而是：

1. 在云 Mac 上生成并编译工程
2. 真机验证亮度、Widget、Intent
3. 补系统桥接方案
4. 对白点值/色彩滤镜做真实桥接或引导闭环
5. 完成签名、审核、测试与商店物料
